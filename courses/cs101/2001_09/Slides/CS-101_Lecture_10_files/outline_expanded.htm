Polymorphism
Two methods with the same signature; which one gets invoked depends on the type of object used to access them.
Interesting only for instance methods where the two definitions are in the same class hierarchy.  That is, there one method must override another for there to be polymorphism.
Assume ClassY is a subclass of ClassX and both define methodA()
ClassX x = new ClassX();
ClassY y = new ClassY();
 x.methodA();
 x = y;
 x.methodA();
Two different methodA()’s get invoked, depending on the actual type of object that x references.

Class Hierarchy
Hello
HelloGUI
HelloCL
HelloBR
Hello has a protected String, msg, which its no-arg constructor initializes
HelloGUI has a no-arg constructor that gets the user’s name using JOptionPane, and builds a different String for msg.
HelloCL has a constructor that receives an array of Strings, which it concatenates with spaces to fill in the user’s name and initialize the msg.
HelloCL has a no-arg constructor that reads the user’s name from the command line.

Class Assignment_03
Provides the public static void main method, which operates as follows:
If args.length is 0, instantiate HelloBR.
If args.length is 1 and if args[0] is “ask”, instantiate HelloGUI.
Otherwise instantiate HelloCL, with args as the parameter passed to the constructor.
After creating the object, print it and exit.

Class BufferedReader
Can be used to read Strings from the user as the program runs.  (Can also read lines from disk files.)
It’s in package java.io, so import java.io.*.
try
{
  BufferedReader br = new BufferedReader(
new InputStreamReader( System.in ) );
  String str = br.readLine();
}
catch ( IOException ioe )
{
  System.err.println( ioe );
  System.exit( 1 );
}
